# Doc Co-Authoring

## Workflow

```text
1. DRAFT   → Produce first version based on requirements
2. REVIEW  → Present for feedback with specific questions
3. REVISE  → Incorporate feedback
4. FINALIZE → Final review, lint, merge
```

## Tone Guide

- **Technical docs**: Direct, precise, example-heavy
- **ADRs**: Context-driven, balanced pros/cons
- **README**: Welcoming, action-oriented, quick start
- **Commit messages**: Imperative, concise, scoped

## Structure Rules

- Start with the "what" and "why"
- Front-load the most important information
- Use tables for comparisons
- Include code examples for every concept
- Keep paragraphs short (3-4 sentences max)

## Feedback Protocol

When presenting drafts, ask:

- "Is the level of detail right?"
- "Should I expand section X?"
- "Does this match the project tone?"

## Markdown Style

- Use ATX headers (`#`, `##`, `###`)
- One sentence per line for diffs
- Code blocks with language specifier
- Tables for structured data
- No trailing whitespace
